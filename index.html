关键产品决策

为什么选 Notion 不自研？

目标用户已在用 Notion，学习成本为零；灵活性满足高度个性化需求。自研工具维护成本远高于在 Notion 上构建。

为什么「变现追踪」是核心卖点？

创作者的痛点不是「学不会笔记」而是「变不了现」。精准匹配目标用户核心诉求。

模板 vs SaaS

MVP 阶段选 Notion 模板——快速验证需求，用户口碑证明后再考虑技术升级。

Vibe Coding — PM 的工程判断力

什么是 Vibe Coding？

用 AI 辅助写代码，自己设计架构、提 Prompt、测试验证、迭代优化。PM 把产品判断力延伸到工程侧。

为什么这对 PM 很重要？

能 Vibe Coding 的 PM 可以自己做 POC，不等开发资源；准确估算工程成本；MVP 速度快 10 倍以上。

这个项目的过程

需求 → 技术草图 → Claude 写 Python 骨架 → 自己测试修 Bug → Streamlit 部署 → 用户反馈 → 迭代。全程约 3 天，零开发成本。

前端：Streamlit

MVP 阶段需要快速验证，Streamlit 让 PM 在不写前端代码的情况下做出可用 Web App。

BiteCard
拍照即记 · AI 抠图卡片 · 热量追踪

iOS 饮食记录 PRD。核心洞察：记录饮食失败不是因为用户不自律，而是工具把「记录结果」做得太丑。BiteCard 把每次记录变成生成精美卡片的仪式感。

1张照生成 BiteCard
4态状态机设计
云端AI识图+抠图
iOSPRD状态机AI识图热量追踪
查看完整 Case Study →

核心产品设计

🎨 抠图卡片（核心差异化）

食物主体 + 盘子整体抠图，4–8px 白色描边。抠图失败 = 记录失败，严苛标准保护了产品差异化。好看的结果本身就是产品功能。

📊 只有一个 kcal 目标

进度环只显示今日 kcal vs 目标。用户只需要知道「今天够了吗」，复杂仪表盘反而是噪音。

🚫 离线强拦截

无网络时完全拒绝创建记录。强约束比「有网再同步」更诚实，避免离线队列的技术复杂性。

查看完整设计 →

四态状态机

State 01
占位
记录已创建，等待云端处理。骨架屏占位，不允许编辑。
State 02
识别中
AI 识图+抠图+营养映射进行中。
State 03
已补全营养
卡片完整展示，计入进度环，可生成分享图。
State 04
失败
任一环节失败。用户主动重试，不自动重试。

「无营养记录」：营养库命中失败时仍允许保存，不计入进度环——先把习惯留住比强求数据完整更重要。

关键技术决策

全部云端，不做端侧推理

端侧模型占存储、更新困难。云端可随时升级模型、AB 测试，无需 App Store 审核。牺牲离线使用与「离线强拦截」原则一致，是有意识的取舍。

Key 不写死在 App 里

所有大模型调用通过自家后端转发——保护 Key 泄露，后端可做限流/计费/模型切换，不用发版 App。

失败路径明确化

识别成功和失败路径完全分离。失败时 errorCode 必填，把后端约束写进 API 规范，不让客户端猜。
