01 · 问题洞察

内容创作者的
最后一公里困境

写好了内容，排版又是一场噩梦

用 Markdown 写完文章，要发小红书就得截图、裁剪、调整，格式全乱；要发公众号就得在编辑器里手动调字体、行距、颜色——写作和排版完全割裂。

多平台内容需要重复排版

同一篇内容，小红书要做成图片卡片，公众号要生成富文本 HTML，是两套完全不同的操作流程，时间大量浪费在格式转换上而不是创作本身。

长内容分页是手工体力活

小红书卡片有尺寸限制（420×560px），长文章需要手动切分多张图，估算哪里断开、每张图放多少字——完全没有规律可循，每次都要重来。

「好内容值得好的呈现形式。排版工具不应该让创作者浪费时间。」
PM 洞察

两个平台，一套 Markdown，一次写作

小红书和公众号是中文内容创作者最重要的两个发布渠道，但它们的排版需求截然不同：小红书需要图片卡片（固定尺寸、视觉驱动），公众号需要富文本 HTML（可读性优先、样式内联）。


把两个工具整合在一个平台里，以 Markdown 作为统一的创作输入，让分发（排版）和创作分离——这才是真正减少创作者摩擦的产品思路。

◎ 核心设计原则

Markdown 是创作语言，平台只负责渲染和导出。创作者只需写一遍，工具负责适配不同平台的展示格式。

02 · 双工具定位

一个平台，
两种发布场景

两个工具在同一平台内，共享 Markdown 输入，面向不同的发布场景。不是功能堆砌，是对「同一内容多平台分发」需求的完整覆盖。

📱 小红书海报生成器

Markdown → 精美图片卡片

以 420×560px（3:4 比例）为标准尺寸，将 Markdown 内容渲染为可直接发布到小红书的高清图片。支持多页自动分页和批量导出 ZIP。

智能分页：自动按字符数分页，或用 ---page--- 手动标记分页点
多页预览：水平滚动查看所有页面，页面导航按钮切换
批量导出：一键生成所有页面图片并打包为 ZIP 下载
样式定制：背景色、字号、颜色、间距全面可调
标准规格：14px 正文，每行 22 汉字，60px 页边距
📝 公众号排版工具

Markdown → 公众号 HTML

将 Markdown 转换为专为公众号优化的带样式 HTML，所有样式内联（适配公众号编辑器不支持外部 CSS 的限制），一键复制后直接粘贴到编辑器。

样式内联：所有 CSS 内联到标签，不依赖外部样式表
实时预览：所见即所得，编辑即时刷新渲染结果
一键复制：复制带样式的完整 HTML，直接粘贴使用
排版优化：专为公众号调整的行距、字号、标题层级样式
03 · 核心功能设计

每个功能都对应
一个真实痛点

功能不是越多越好。MarkFlow 的每个功能决策都从「创作者在哪个环节卡住了」出发，不做锦上添花，只做雪中送炭。

智能自动分页

根据内容字符量自动估算最优分页点，提供 5 档内容量设置（300–3000字/页）。再也不用手动估算每张图能放多少字。

手动分页标记

在 Markdown 中插入 ---page--- 即可精确控制分页位置，与自动分页模式互补，给创作者完全的控制权。

批量导出 ZIP

多页模式下，一次性把所有页面渲染为高清 PNG（2x 缩放），自动打包成 ZIP 下载。5 页内容的导出，从 10 分钟截图变成 10 秒点击。

CSS 样式内联

公众号编辑器不支持 <style> 标签。工具自动将所有样式内联到每个 HTML 元素，粘贴后直接可用，不会丢失格式。

实时双面板预览

左侧 Markdown 编辑，右侧实时渲染结果。编辑器输入即时同步预览，不需要手动触发渲染，所见即所得。

样式主题定制

背景色（纯色/渐变）、正文字号、文字颜色、内边距行距均可调整。保证每个人都能找到适合自己内容风格的视觉方案。

05 · 技术方案

端侧全量运行，
无服务器依赖

◎ 两套 Markdown 解析器，各司其职

海报生成器用 markdown-it——灵活、插件丰富，适合自定义渲染成卡片 HTML；公众号排版用 marked——专注输出标准 HTML，配合样式内联更稳定。两个场景需求不同，选择最合适的工具而不是强求统一。

◈ html2canvas 的 2x 缩放决策

小红书内容通常在高分屏上查看，1x 分辨率的 PNG 会显得模糊。html2canvas 导出时用 2x 设备像素比，输出 840×1120px 高清图（显示为 420×560px），确保每张卡片在移动端看起来清晰锐利。

06 · 产品思考

做完这个工具
让我想清楚了什么

「整合」比「各自做好」更有价值

小红书海报工具有很多，公众号排版工具也有很多——但把两个工具以 Markdown 为纽带放在同一个界面，减少了「切换工具」的摩擦。整合本身就是产品价值，而不只是功能叠加。

给用户控制权，同时给足智能兜底

自动分页负责效率（大多数时候用户不需要手动干预），手动 ---page--- 标记负责精确控制。这种「智能默认 + 精确覆盖」的设计思路，适用于绝大多数有「自动化 vs 控制权」矛盾的功能设计场景。

无后端不等于功能弱，等于部署成本极低

整个工具在浏览器本地运行，Markdown 解析、图片生成、ZIP 打包全在前端完成，不需要任何服务器。这意味着：任何人可以在 GitHub Pages / Netlify / Vercel 上免费部署，维护成本几乎为零。

在 MVP 阶段，「能快速让用户用到」比「架构优雅」更重要。选择纯前端方案是正确的性价比判断。

分页算法是产品决策，不只是技术问题

每页 600 字符（默认小档）这个数字，背后是「14px 字号 × 每行 22 汉字 × 420px 卡片宽度 × 560px 卡片高度 × 60px 页边距」的计算结果，是对「一张卡片能放多少内容还不显得拥挤」的产品判断——不是随机数字，是从用户体验出发的参数设计。

从写完到发出，
不应该再有障碍。

MarkFlow 解决的不是「怎么写」的问题，而是「写好了之后怎么高效发出」的问题。双工具整合、智能分页、批量导出、CSS 内联——每一个功能都对应一个创作者会卡住的真实场景。
